Ch. 04 · Method

Five rules I actually enforce.

Working principles, not slogans
RULE I· senior-architect first

Plan at the architecture level before writing a line.

The first deliverable on any non-trivial task is a diagram and a build sequence — not a pull request. Code is the cheapest part; the expensive part is reversing a wrong blueprint six weeks in.

RULE II· real only

Never mock. Always hit the real service.

Mocks lie. A test that passes against a mock and fails against production is worse than no test. I wire against real APIs, real databases, real messaging — and I verify via MCP browser tools before declaring done.

RULE III· fail loud

Silence ships false negatives to customers.

No except: pass. No swallowed return-empty-list. Every suppressed error either raises, writes to result.error, or logs a WARNING "COVERAGE GAP". Users deserve to know what's missing, not what's convenient.

RULE IV· pass, fail, or delete

A skipped test is a lie in the CI dashboard.

Zero tolerance for skip, xfail, or "temporarily disabled". A test either passes, fails and gets fixed, or gets deleted. Green should mean green — anything else is trust debt compounding silently.

RULE V· evidence before assertion

"Done" is a claim. Claims require proof.

I don't say "it works" — I run the command, read the output, screenshot the UI, and paste the evidence. Verification before completion is an iron law: green logs and a live URL, or the work isn't finished.

RULE VI· own the full arc

Architecture → code → infra → docs → on-call.

I don't throw work over a wall. If I shipped it, I can diagram it, deploy it, debug it at 03:00, and write the documentation a new hire can read on day one. That's what senior actually means.
